Getting Started With Gulp

What is Gulp?

Gulp is an automatic task runner, or build system, written in node.js and can handle many workflow tasks. In his git-book, David Nowinsky described Gulp as a factory. I really like this description. Gulp, much like a factory, takes files and turns them into other files.

You might question why this is needed, and to some it might not be. If you’re working on a simple site written in pure CSS, then it most likely not needed. However, if you’re doing a lot of mundane tasks, you could benefit from using Gulp. Build systems can watch files for changes, compile to CSS or Javascript, upload files to servers, optimize images, concatenate, minify, and much more.

Installation

Installation for mac is pretty simple, however there are a few dependencies. First off, you will need to install node.js. Once you've ran through the installation process for node.js, we will install the Node Package Manager, or NPM, through the Console. Open the console and type curl -L https://npmjs.org/install.sh | sh after the $. You will have to install NPM as a root user, so you will probably have to throw sudo in front of that command. Once you have Node and NPM ready, run npm install -g gulp to install Gulp globally.

Gulp Basics

Plugins

Plugins extend the functionality of Gulp. Gulp's philosophy is that plugins should only perform one action, this way it is easy to combine plugins to create complicated functionality easily. All approved plugins are listing on Gulp's plugin page. Gulp curates their plugins to ensure high quality. It's suggested you use the official Gulp plugin database to ensure you do not use harmful or non-working plugins.

Streams

Gulp uses streams to link programs, or in this case plugins, together. This relieves your system of temporary files while providing blazing fast functionality. Going back to David Nowinsky's factory analogy, they are the conveyor belts, moving from one program to the other.

Tasks

Tasks are how you control Gulp and are defined with gulp.task function. Tasks can do many things, including interacting with plugins and streams. You can link tasks together and create dependencies. If streams are the conveyor belt, then tasks are the assembly line.

Using Gulp

Now that you have your feet wet in Gulp, it's time to start using it. We will mostly be using the console for this part, however feel free to open and edit code in your favorite text editor.

Creating a Project

First things first, we're going to need a new project. In your console, enter the following to create a new project folder mkdir my_gulp_project. This will create a folder inside your home directory. mkdir stands for "make directory", or folder. To open that directory inside the console, we're going to use the cd, or "change directory", command. Type the following cd my_gulp_project.

Once inside my_gulp_project, we're going to use the "npm init" to create our devDependencies. This command will create a package.json file and a node_modules directory. These are important when sharing our project. If the devDependencies are up to date, a fellow developer would only need to download your package.json file and type npm install into the terminal to download all of the plugins used with this project. You should never update the package.json file or the node_modules directory. That is NPM's job. So, without further ado, type npm init in your console. You'll get a few prompts asking for information, but none of it is required or important for us, so simply hit enter until they're gone.

devDependencies

I know what you're thinking. "I've already installed Gulp!", and you are correct. However, you've installed it globally and it's necessary to include it inside your devDependencies. In your console, type in npm install --save-dev gulp. You should notice a difference from the Gulp install command you ran earlier, that being the addition of the "--save-dev" flag. This flag let's NPM know you'd like to save this as a dependency.

Next, we're going to create a gulpfile.js inside of our directory. This file is how we will be telling Gulp what to do and when to do it. In your console, type touch gulpfile.js. The "touch" command will create the file. Now open gulpfile.js in your editor, and on line 1 paste the following: var gulp = require('gulp');.

Plugins and Tasks

We're going to use Gulp's official plugins list to find and install our first plugin. I like to code in Sass, so I will search for a Sass compiler. The query returned gulp-ruby-sass, which looks exactly like what I need. Gulp-ruby-sass does have a two dependencies, Ruby and Sass. If you're on a Mac, Ruby comes already installed! To install Sass, all you need to do is run gem install sass inside your console.

Now that we have Ruby and Sass, it's time to install gulp-ruby-sass. Run npm install --save-dev gulp-ruby-sass in your console. This will install gulp-ruby-sass as a devDependency inside both package.json and node_modules.

Open gulpfile.js and edit it until it looks like something below.

// Requirements
var gulp = require('gulp'),
sass = require('gulp-ruby-sass');
// Defines path to sass
var sassRoot = 'build/sass/';
// Gulp task
gulp.task('sass-to-css', function(){
return gulp.src(sassRoot+'main.scss')
.pipe(sass({sourcemap: true}))
.pipe(gulp.dest('public/css'));
});

Requirements

Requirements are basically just declaring your build systems dependencies. I won't get into this too much, as they're pretty self explanatory. If you do not include your plugin in your requirements, Gulp will not run it.

Defines path to sass

This is optionally and really just a personal preference. You do not have to declare variables like this for Gulp to work, I just think it's neat.

Gulp task

I mentioned Gulp tasks earlier, but I will go into detail here. Tasks are the assembly line of your factory. It will take the source, return gulp.src(sassRoot+'main.scss'), run it through streams, .pipe(sass({sourcemap: true})), and the final product will arrive at the destination, .pipe(gulp.dest('public/css'));. As mentioned, you can link multiple plugins into a task to form super-tasks. That's not an official name, I just like it. Anyway, we will get more into this later on in the post.

Running Gulp

You can run this task by entering gulp sass-to-css, however since there isn't a Sass file, or even a Sass or build folder, this command will out put nothing. So, let's create those folders and files! In your console, write mkdir -p build/sass. You probably recognize mkdir, but you might not recognize the -p, or --parent, flag. Typically, mkdir will not make nested directories. The -p flag will tell it to make the parent directories as well. Once the directories are created, run touch build/sass/main.scss to create our Sass file. Now let's open main.scss in our favorite editor and insert the following:

// Just a test
body{
background: black;
}

Save the file and return back to the terminal. Now that we have the folder, file, and a simple code, let's run that Gulp command. Enter gulp sass-to-css and hit enter. If you did everything correctly, you should and output like below from the console:

[16:33:57] Using gulpfile ~/projects/test/gulpfile.js
[16:33:57] Starting 'sass-to-css'...
[16:33:57] Finished 'sass-to-css' after 389 ms

Much of this output is arbitrary, but the important part is the last line. It let's you know the Gulp task is finished. If you enter the ls command into your console, which stands for "list" and will list everything in the directory, you will notice there is now a "public" directory. Inside that public directory, there should be a css directory and inside that, there is a main.css file. Congrats, you've successfully ran a Gulp command!

Super tasks

Now that we have the basics down, let's create one of those super-tasks I mentioned before. Since we focused on CSS last time, let's stick with it. One thing I absolutely detest writing are vendor prefixes. They take my otherwise nice and simple code and turn it into a bloated monster. So, let's add an auto-prefixer to our workflow. by searching the Gulp plugin database, I've came across the gulp-autoprefixer, which looks like it does precisely what we'd like. First step is installation. In your console, enter npm install --save-dev gulp-autoprefixer

Next, we're going to update our gulpfile.js to include gulp-autoprefixer. Your requirement sections should now look like the code below:

// Requirements
var gulp = require('gulp'),
sass = require('gulp-ruby-sass'),
prefix = require('gulp-autoprefixer');

And we'll update our tasks to the following as well:

// Gulp task
gulp.task('sass-to-css', function(){
return gulp.src(sassRoot+'main.scss')
.pipe(sass({sourcemap: true}))
.pipe(prefix("last 3 versions"))
.pipe(gulp.dest('public/css'));
});

Inside the new pipe, you'll see "last 3 versions". Sometimes plugins have additional options. This is basically telling the auto-prefixer to get the last 3 versions of browsers.

Let's test this out. Open your main.scss and edit the code to the following:

// Just a test
body{
background: black;
}
a{
transition: .3s;
}

Now in your console, run gulp sass-to-css. Now open main.css. It should look like something below:

// Just a test
body{
background: black; }
a{
-webkit-transition: .3s;
transition: .3s; }

And there is your Super-task!

Bonus

Here are some helpful tips on how to get the most from Gulp.

Gulp watch

Gulp is able to watch your files for any changes and run a task right out of the box. This is awesome for many reasons that are self explanatory. To have gulp watch, simply create a new task for it.

gulp.task('watch-sass', function(){
// What to watch
gulp.watch('build/sass/main.scss', function(){
// What to run
gulp.run('sass-to-css');
});
});

// What to watch

This is your file structure. Right now, it is pointing to /build/css/main.scss, because sass-to-css uses that file as a source.

// What to run

Here is where we include our task we'd like to run, in this case, sass-to-css. However, it can be any task you fancy.

All Files

Above, we targeted the single file of "build/sass/main.scss", but what if we had many files? or many imports into the Sass file? Well, Gulp has our back. By changing "build/sass/main.scss" to "build/sass/*.scss", Gulp will run the tasks on all files inside "build/sass/". If you have many sub-directories, simply use the following to target all files in all sub directories "build/sass/**/*.scss".

Using Tasks as dependencies

You can also make tasks require to run other tasks. This is useful if you'd like to keep a task to be ran separate, but also need it for a larger task.

// Gulp task
gulp.task('sass-to-css', function(){
return gulp.src(sassRoot+'main.scss')
.pipe(sass({sourcemap: true}))
.pipe(gulp.dest('public/css'));
});
gulp.task('prefix', ['sass-to-css'], function(){
return gulp.src('public/css/*.css')
.pipe(prefix("last 3 versions"))
.pipe(gulp.dest('public/css'));
});

In the code above, prefix is require sass-to-css in order to run. You can include multiple tasks requirements, but they won't occur in order. A better option is to nest requirements, like below:

// Gulp task
gulp.task('sass-to-css', function(){
return gulp.src(sassRoot+'main.scss')
.pipe(sass({sourcemap: true}))
.pipe(gulp.dest('public/css'));
});
gulp.task('prefix', ['sass-to-css'], function(){
return gulp.src('public/css/*.css')
.pipe(prefix("last 3 versions"))
.pipe(gulp.dest('public/css'));
});
gulp.task('minify-css', ['prefix'], function(){
return gulp.src('public/css/*.css')
.pipe(minify())
.pipe(gulp.dest('public/css'));
});
